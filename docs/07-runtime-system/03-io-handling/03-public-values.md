# Public Values

## Overview

Public values are the shared data elements that both prover and verifier agree upon in a zero-knowledge proof. They form the public statement that the proof attests to, representing the inputs, outputs, or intermediate commitments that define what is being proven. While the proof hides the details of computation, public values establish the common ground between prover and verifier.

Understanding public values is essential for designing zkVM applications. They determine what information is visible, what claims the proof makes, and how verification works. Public values bridge the gap between the private computation inside the zkVM and the public claims that emerge from it.

This document covers the role of public values, their handling in zkVM systems, and design considerations for effective use.

## Public Value Fundamentals

### Definition

What public values are:

```
Public values:
  Data known to both prover and verifier
  Part of the proof statement
  Not hidden by zero-knowledge

Examples:
  Input commitments
  Output results
  Merkle roots
  Timestamps
```

### Role in Proofs

Function in zero-knowledge proofs:

```
Public values define:
  What is being proven
  Shared reference points
  Verification inputs

Without public values:
  No verifiable claim
  Nothing to check against
  Meaningless proof
```

### Public vs Private

Distinguishing public and private:

```
Public values:
  Visible to all
  Part of proof statement
  Verification input

Private values (witness):
  Known only to prover
  Hidden in proof
  Not revealed
```

## Commitment Schemes

### Hash Commitments

Simple hash-based commitment:

```
Hash commitment:
  H(value) = commitment
  Reveal value to verify
  One-way binding

Properties:
  Simple
  Efficient
  Hiding until reveal
```

### Merkle Commitments

Tree-based commitment:

```
Merkle commitment:
  Hash tree of values
  Root is commitment
  Partial reveal possible

Benefits:
  Large data sets
  Selective disclosure
  Efficient updates
```

### Polynomial Commitments

Algebraic commitment:

```
Polynomial commitment:
  Commit to polynomial
  Open at points
  Used in STARK/SNARK

Role:
  Trace commitment
  Constraint commitment
  Query responses
```

## Design Trade-offs

### Privacy vs Verifiability

| More Privacy | More Transparency |
|--------------|-------------------|
| Commit only | Reveal values |
| Less verifier info | Full verifier info |
| Trust commitments | Direct verification |
| Complex claims | Simple claims |

### Size vs Expressiveness

| Minimal Public | Rich Public |
|----------------|-------------|
| Fewer values | More values |
| Smaller proofs | Larger proofs |
| Less direct info | More direct info |
| Commitment-based | Value-based |

### Fixed vs Flexible

| Fixed Format | Flexible Format |
|--------------|-----------------|
| Predictable | Adaptable |
| Simpler verification | Complex verification |
| Less expressive | More expressive |
| Standard | Custom |
