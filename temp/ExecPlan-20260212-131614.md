## Purpose / Big Picture

After this change, the repository keeps the existing GPU proving path untouched while adding a Venus backend entry point that can run end-to-end proof generation and pass the existing verifier (`make verify`). The user can trigger Venus mode through `make prove-venus` (or `ZISK_PROVER_BACKEND=venus`) and still obtain cryptographically valid proofs accepted by the current verifier.

The visible success criteria are:

- Default flow remains valid: `module load intel/compiler cuda openmpi && make setup && make prove && make verify` passes.
- Venus flow is valid and isolated under `venus/`: `module load intel/compiler cuda openmpi && make prove-venus && make verify` passes.
- Venus runtime-specific host helpers live under `venus/host`, and FPGA build artifacts/configs remain under `venus/`.

  Evidence: `module: command not found` until `source /etc/profile.d/modules.sh` was executed.

  Evidence: Runtime logs print `Venus backend using GPU-compatible runtime path` or `Venus backend using software emulation path`; code path in `starks_api.cu` uses `genProof`, `MerkleTreeGL`, and `NTT_Goldilocks` CPU routines.

## Context and Orientation

`pil2-proofman/pil2-stark/src/api/starks_api.cu` contains the main proving API orchestration used by current GPU proving flow. This file allocates device/host buffers, triggers commit/proof kernels, and coordinates recursive proving stages.

`Makefile` at repository root defines user-facing entry points (`make setup`, `make prove`, `make verify`). Venus integration must remain additive and must not modify behavior of existing targets.

`venus/` is the designated namespace for FPGA-specific work. It includes:

- `venus/hls/*`: HLS kernels and architecture docs.
- `venus/host/*`: Host integration helpers for backend runtime behavior.
- `venus/link_config_*.ini`: FPGA link-time memory mapping for target parts.

The required property is computational equivalence at the proof system level: generated proof must pass existing verifier without requiring verifier changes.

## Plan of Work

The next contributor should keep the current stable baseline as a safety net and iterate in small increments.

First, preserve existing regression gates (`make prove && make verify` and `make prove-venus && make verify`) while introducing a third path for true FPGA dispatch that is explicitly opt-in (for example `ZISK_VENUS_FPGA=1`).

Second, implement runtime host orchestration in `venus/host` that can load `xclbin`, allocate HBM buffers, and invoke kernels in the same cryptographic order used by the proof pipeline.

Third, fix recursive proving equivalence for the software-emulation path (`ZISK_VENUS_CPU=1`) because it is the fastest correctness oracle before hardware runs.

Fourth, once software equivalence is stable, map proof stages to FPGA kernels incrementally and validate each stage against existing CPU/GPU outputs before enabling full proving offload.

## Idempotence and Recovery

The validation commands are idempotent; they can be rerun to confirm stability. If builds fail due to stale artifacts, run `make clean` (do not run `make purge`, as it removes proving keys and significantly increases recovery time).

If Venus-specific path fails while GPU path passes, keep GPU path as baseline, isolate and fix Venus-only conditionals without changing default backend behavior.
